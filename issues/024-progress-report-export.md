# 024 — Progress Report Export [AFK]

**Type:** AFK  
**PRD User Stories:** 56, 57

## What to build

Doctors can export a structured student progress report as a print-ready view. The API returns a complete JSON payload (all longitudinal data in one response); the frontend renders it into a print-optimised layout using CSS `@media print`. The report includes the student snapshot, assessment history, goal trajectories, sensory deltas, session attendance, and plan history. Per-tenant branding (logo on reports) is behind a `per_tenant_branding` feature flag.

## Acceptance criteria

**Schema / migrations**
- [ ] No new tables

**API endpoints**
- [ ] `GET /students/:id/progress-report` — doctor or clinic_admin: returns a single structured JSON payload containing:
  - Student demographics snapshot (name, DOB, age, OP number, school, guardians)
  - All finalised assessments (Form 1s) with diagnoses, milestones, sensory profile, goals, intervention plan
  - All follow-up assessments (Form 2s) with goal reviews, sensory re-ratings, plan adjustments
  - Goal trajectories (full `GoalProgressEntry` timeline per goal)
  - Session summary: total scheduled, attended, missed, attendance %
  - Treatment plan history: all versions with game lists and status changes
  - Report metadata: `generatedAt`, `generatedByUserId`, `clinicName`, `clinicLogoUrl` (null if feature flag off)
- [ ] `clinicLogoUrl` is included only if `SubscriptionPlan.feature_flags.per_tenant_branding: true` for the clinic

**Frontend**
- [ ] "Export Progress Report" button on the student detail page (doctor/clinic_admin only)
- [ ] Clicking opens a new browser tab with `/students/:id/progress-report-view`
- [ ] Report view page (`/students/:id/progress-report-view`): fetches `GET /students/:id/progress-report`, renders a structured HTML layout:
  - Cover section: clinic logo (if feature flag enabled), clinic name, report title, student name, report generation date
  - Section 1: Patient demographics and guardian info
  - Section 2: Assessment history (one sub-section per Form 1 version)
  - Section 3: Goal progress (tables + mini sparklines)
  - Section 4: Sensory profile change table (baseline vs. each follow-up for all 7 systems)
  - Section 5: Session attendance summary (table by month)
  - Section 6: Treatment plan history
  - Footer: "Generated by HaberApp on [date]"
- [ ] Print button (and `window.print()` triggered via keyboard shortcut `Ctrl+P`) → CSS `@media print` hides navigation, shows only the report content
- [ ] Report is not editable — read-only rendering

**Tests**
- [ ] `GET /students/:id/progress-report` returns all required top-level keys
- [ ] Report is not accessible by a therapist not assigned to the student (403)
- [ ] `clinicLogoUrl` is null in the response when `per_tenant_branding: false`
- [ ] `clinicLogoUrl` is a non-null URL when `per_tenant_branding: true` (integration test with a seeded clinic logo URL)
- [ ] Goal trajectories in the report match `GET /goals/:goalId/trajectory` for each goal
- [ ] Report for a student with no follow-ups omits the Section 4 sensory change table gracefully

## QA / Manual testing

- [ ] Log in as doctor → open a student with at least 2 assessments, 2 follow-ups, and 5+ sessions → click "Export Progress Report"
- [ ] Verify a new tab opens with the print-ready report layout
- [ ] Verify the cover page shows clinic name and "Generated by HaberApp" footer
- [ ] Verify Section 3 shows correct goal progress percentages matching what was entered in follow-ups
- [ ] Verify Section 4 sensory change table shows the initial baseline and follow-up deltas
- [ ] Press `Ctrl+P` → verify the browser print dialog opens with only report content visible (no sidebar, no nav)
- [ ] Enable `per_tenant_branding` in the clinic's subscription plan → refresh the report → verify the clinic logo appears in the cover section
