Core MVP Ready

The core clinical workflow (intake → assessment → plan → sessions → follow-up) is fully built end-to-end. The main blockers for a demoable MVP are dashboards & reporting (all three planned as FE-09/FE-10/FE-11 + BE-14), the review-due indicator for multi-therapist coverage, and minor schema/permission gaps listed below. Test suites (TEST-01–04) are filed but unimplemented — high risk for a clinical application.

Gap Analysis — Critical Blockers for Demo MVP

GAP 1: Dashboards & Reporting (All 3 views missing)
HIGH   Blocks: US#45, US#46, US#48  ·  Planned Issues: BE-14, FE-09, FE-10, FE-11

The per-child longitudinal dashboard, clinic admin operational dashboard, and progress report are entirely unimplemented. Data exists in the DB but the aggregation tRPC procedures (dashboard.*, report.*) are not built. For a clinical demo this is critical — therapists and clinic admins have no visibility into progress.

  • BE-14 must be built first (6 dashboard procedures + 1 report procedure)
  • FE-09 (per-child dashboard with Recharts) depends on BE-14
  • FE-10 (clinic + super admin dashboards) depends on BE-14
  • FE-11 (progress report print view) depends on BE-14
GAP 2: Review-Due Indicator & Claim Flow (Multi-Therapist)
HIGH   Blocks: US#53, US#54  ·  New Issue: FE-12

The DB schema (ChildTherapistAssignment.reviewDueAt / reviewClaimed / reviewClaimedBy) and the intake form (sets reviewDueAt) are in place. But there is no:

  • Badge/indicator on the child list or any other screen showing "review due"
  • API procedure to claim a review (set reviewClaimed=true, reviewClaimedBy=therapistId)
  • FE flow where opening a follow-up auto-claims the review slot
GAP 3: Guardian Consent Model — 1:1 vs Unanimous Multi-Guardian
HIGH   PRD Mismatch  ·  New Issue: BE-16

PRD section says "unanimous guardian consent model" — if any guardian withdraws, consent is withdrawn for all. But the DB has Guardian.childId @unique, meaning one child can only have one guardian record. ConsentRecord is per-child (not per-guardian). The unanimous model cannot work with only one guardian.

This requires a schema change: remove @unique on Guardian.childId, allow multiple Guardian rows per child, and update consent withdrawal logic to check all guardians.

GAP 4: Missing Permission — treatment_plan.modify_minor
MEDIUM   Blocks: US#10  ·  New Issue: BE-17

Only 3 permissions defined in packages/db/src/permissions.ts. PRD US#10 explicitly names a 4th: treatment_plan.modify_minor — to grant senior therapists minor plan edits without full plan-modify rights. No route currently enforces this permission.

GAP 5: Child Photo Upload UI
MEDIUM   Blocks: US#11  ·  New Issue: FE-13

Child.photoUrl field exists in the DB schema, but the 5-step child intake wizard has no file upload component. A photo upload (to S3/R2/Supabase Storage) with the URL stored in photoUrl is needed.

GAP 6: Admin UI for Child Soft-Delete
MEDIUM   Blocks: US#16, US#77  ·  New Issue: FE-14

child.softDelete procedure exists but there is no UI for clinic admins or super admins to trigger it. The child profile and child list pages do not expose a delete action. Required for DPDP compliance demonstration.

GAP 7: Assessment Section G Goals Not Linked to Goal Entity
MEDIUM   Blocks: US#61  ·  Recommendation: Fold into BE-17

When a therapist fills in Section G of the initial assessment (up to 4 short-term + 4 long-term goals), those goals are stored as free text inside sectionG JSONB. They are not automatically created as first-class Goal entities on the treatment plan. This breaks the longitudinal goal-tracking flow where GoalProgressEntry records are appended at follow-ups.

Recommendation: when assessment.create is called with Section G data, auto-create Goal rows on the child's active or newly created plan.

GAP 8: Zero Automated Test Coverage
LOW (risk)   Planned Issues: TEST-01, TEST-02, TEST-03, TEST-04

No integration tests exist for any module. For a clinical app handling child health data, this is a significant risk. Tests are planned across 4 suites covering AuthService, ConsentService, AssessmentService, GoalService, SessionGenerator, GameResultService, and TreatmentPlanService.

GAP 9: plan.modify Missing ADD Goal Action
LOW   Minor workflow gap

plan.modify accepts CLOSE / CARRY_OVER / MODIFY for existing goals but does not accept an ADD action to create brand-new goals in the same transaction. PRD US#71 lists "add" as one of the four decisions. Workaround: call goal.create separately after modify. This is functionally equivalent but creates an inconsistent UX — the plan modify wizard should include a "new goal" section.

Architecture Notes

AreaObservation
Session generation Synchronous on plan activation — no background job. For a 52-week plan at 3x/week, this creates 156 session rows in one transaction. Should be acceptable for V1 but watch for timeout on large plans.
Permissions model Currently all-or-nothing per string. The UserPermission table supports arbitrary strings, making the modify_minor addition a zero-schema-change addition. Only permissions.ts and route checks need updating.
Assessment JSONB vs relational Sections A–H stored as JSONB blobs. This is pragmatic for the complex form structure but makes querying specific sub-fields (e.g. "show all children with Tactile rating ≥ 4") impossible without JSON path queries. Acceptable for V1; sensory data that needs querying is already in the relational SensoryProfile table.
Plan phases TreatmentPlan.phases is JSONB (no separate Phase model). PRD says "optional phases (e.g., 4 cycles of 2 weeks)". This is fine for V1; PlanGameAssignment.appliesToPhase references phase index by convention.
tRPC vs webhooks Game webhook endpoints are plain Hono HTTP routes (not tRPC) — correct per PRD design. All other client–server communication is tRPC. Split is clean.
Soft-delete filtering Currently child.list appends deletedAt: null manually in the query. BE-15 adds a Prisma middleware to enforce this globally. Until BE-15 ships, any new query touching Child must remember to add the filter.
